# 智慧化密碼管理器 — 規格書 (Technical Specification)

**版本**：v1.1
**日期**：2026-06-15
**目標平台**：PWA（手機 + 電腦，單一程式碼庫、可離線）
**對應需求書**：`需求書.md`
**本版變更**：新增 §9 跨裝置端對端加密同步（金鑰派生關係、密文同步流程、衝突合併）與 §10 帳號復原（復原碼機制與忘記主密碼流程）；§5、§6 補充對應說明。

## 1. 系統架構總覽

採用 **本地優先 + 可選雲端增強（Hybrid Architecture）**。密碼相關運算全部在 client（PWA）完成；Server 為可選元件，只接觸非敏感資料。

```
┌──────────────────────────── PWA (手機 / 電腦) ────────────────────────────┐
│  UI 層        匯入確認卡片 | 搜尋框 | 條目列表 | 設定                       │
│  邏輯層       Parser pipeline | 別名/模糊搜尋 | 加解密 | (可選) 本地embedding│
│  儲存層       加密本地 DB (SQLite + SQLCipher / 加密 payload)               │
│  金鑰         主密碼派生金鑰，僅存記憶體 / 本地，永不上傳                    │
└────────────────────────────────────┬───────────────────────────────────────┘
                                      │ 只傳「非敏感」資料（可選）
                                      ▼
┌──────────────────────────── Server (可選) ────────────────────────────────┐
│  embedding 服務 | 向量索引 / 檢索   （不存 password、不存解密金鑰）          │
│  回傳：只有條目 ID 清單                                                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

設計原則：
- **搜尋**可在 Server 做，但只回傳 ID，client 再用 ID 在本地解密取出密碼。
- **Server 永遠看不到** password / username（可選不傳或雜湊）/ otp / note。
- AI 能力為漸進增強：關閉雲端與本地模型時，別名 + 模糊搜尋仍完整可用。

## 2. 資料模型

### 2.1 核心結構

一個 service 可包含多組 credentials；username 與 password 皆為選填。

```ts
interface ServiceEntry {
  id: string;                 // e.g. "abc123"
  service: string;            // "Facebook"
  aliases: string[];          // ["fb", "臉書", "facebook"]
  url?: string;               // "facebook.com"
  tags?: string[];
  credentials: Credential[];
}

interface Credential {
  username?: string;          // 選填
  password?: string;          // 選填
  otp?: string;               // 選填
  note?: string;              // 選填
}
```

範例（ProtonVPN 多組帳號）：

```json
{
  "id": "abc123",
  "service": "protonvpn",
  "aliases": ["proton", "vpn"],
  "credentials": [
    { "username": "nnchen99", "password": "nbchen11" },
    { "username": "ncchen99", "password": "nbchen11" }
  ]
}
```

### 2.2 本地儲存 vs 雲端儲存（職責分離）

本地（加密）：

```json
{ "id": "abc123", "encrypted_payload": "..." }
```

雲端（可選，僅索引）：

```json
{ "id": "abc123", "embedding": [/* 由非敏感欄位產生 */] }
```

`encrypted_payload` 解密後即為上方 `ServiceEntry`。雲端僅持有 `id` 與向量，無從還原內容。

### 2.3 搜尋用非敏感欄位

僅對下列欄位建立索引 / embedding，**絕不**包含 password / otp / note：

```json
{ "search_text": "facebook fb 臉書 social meta facebook.com" }
```

## 3. 智慧化匯入：Parser Pipeline

解析的重點不是「理解自然語言」，而是「建立容錯率高的正規化 + 啟發式管線」。

### 3.1 Step 0 — Input Canonicalization（輸入正規化）

在解析前先把雜亂格式統一，後續 parser 才會簡單。

1. **欄位同義詞字典**

```ts
const FIELD_ALIASES: Record<string, "username" | "password"> = {
  account: "username", acc: "username", 帳號: "username",
  帳戶: "username", user: "username",
  pwd: "password", pass: "password", password: "password", 密碼: "password",
};
```

2. **大小寫正規化**：`ACC` → `acc`
3. **全形 / 半形轉換**：`：` → `:`、全形連字號 → `-`
4. **統一分隔符號**：`：` `-` `=` `>` `】` `|` 等 → 統一成 `:`

結果：`ACC：abc` → `acc:abc`、`帳號-abc` → `username:abc`

### 3.2 Step 1 — Line Classification（逐行分類）

```ts
type LineType = "service" | "username" | "password" | "unknown";
```

以 regex 比對正規化後的關鍵字，例如：

```
^(acc|account|帳號|帳戶|user)\s*:   → username
^(pwd|pass|password|密碼)\s*:        → password
```

### 3.3 Step 2 — Heuristic Scoring（特徵推論）

當行首沒有明確欄位關鍵字時，依特徵打分判斷：

| Pattern | 推論欄位 | Score |
|---|---|---|
| email（含 @） | username | +10 |
| phone（如 09xxxxxxxx） | username | +8 |
| 含 `.com` / URL | service / url | +10 |
| 大小寫 + 數字混合、高 entropy | password | +7 |
| 含特殊符號 | password | +5 |

採 probabilistic parsing：取最高分作為該行欄位，並保留 confidence。

### 3.4 Step 3 — FSM Grouping（有限狀態機分組）

```ts
let currentService: ServiceEntry | null = null;
let currentCredential: Credential | null = null;
```

規則：
- 遇 service 行 → 開新 `ServiceEntry`。
- 遇 username / password 行 → 寫入 `currentCredential`（若不存在則新建）。
- 遇**空行** → commit 當前 credential，下一組 credential 重新開始（支援同 service 多組帳密）。
- 遇只有一行數字 + 一行高 entropy 字串（無關鍵字）→ 第一行偏 username、第二行偏 password。

範例輸入：

```
ProtonVPN
acc: nnchen99
pwd: nbchen11

acc: ncchen99
pwd: nbchen11

藝fun
0906516870
Nbchen11
```

解析輸出：`protonvpn`（兩組 credentials）＋ `藝fun`（username 0906516870 / password Nbchen11，由特徵推論）。

### 3.5 Step 4 — Confirm UI（AI Assist, not AI Auto）

解析結果以可編輯卡片呈現，使用者逐筆確認後才寫入。提高容錯率與信任：

```
偵測到：
Service: ProtonVPN
  Credential 1   Username: nnchen99   Password: ********
  Credential 2   Username: ncchen99   Password: ********
[ 確認 ]  [ 編輯 ]  [ 捨棄 ]
```

目標解析正確率 85%+，其餘由此步驟人工修正。

## 4. 搜尋設計

### 4.1 預設：別名 + 模糊搜尋（不需 LLM）

- 正規化查詢（lowercase、去空白）。
- 對 `service` + `aliases` + `url` + `tags` 做 trigram / Levenshtein 模糊比對。
- 別名字典可自動產生（由 domain 推導：`facebook.com` → `facebook`, `fb`, `臉書`）並允許手動維護。

```json
{
  "facebook": ["fb", "臉書"],
  "instagram": ["ig", "哀居"],
  "messenger": ["mes", "訊息"]
}
```

技術選型：**SQLite FTS5 + trigram + alias table**。資料量數十～數千筆即可即時回應，通常不需要 vector DB。

### 4.2 可選：語意搜尋（小型 embedding 模型）

需要時使用 **embedding 模型而非生成模型**，差距極大：

- 候選：MiniLM / bge-small / e5-small，量化後約數十 MB（相對 WebLLM 的數 GB）。
- 僅對非敏感 `search_text` 產生向量。
- 可在本地（瀏覽器內）或在 Server 端運算。

### 4.3 隱私保護的雲端搜尋流程

```
使用者輸入 "FB"
  → Client 將查詢字串送 Server embedding
  → Server 在向量索引檢索，回傳 ["abc123"]（只有 ID）
  → Client 依 ID 取本地 encrypted_payload 解密 → 顯示帳密
```

Server 全程不知道 password / username / note。

## 5. 安全與加密

- **本地加密**：SQLite + SQLCipher，或自管 `encrypted_payload`（AES-256-GCM）。
- **金鑰派生**：由主密碼經 KDF（建議 Argon2id，次選 PBKDF2）派生，金鑰僅存記憶體 / 本地，永不上傳。完整金鑰階層見 §9.1。
- **上傳資料最小化**：啟用雲端搜尋時，僅上傳 `id` + 非敏感欄位的 embedding/索引；password / otp / note 一律不上傳；username 可選不傳或雜湊化；URL 可雜湊化。
- **端對端加密同步**：啟用同步時，雲端僅持有密文（見 §9），採零知識架構，伺服器無法解讀。
- **離線優先**：核心儲存與搜尋（別名 + 模糊）完全離線可用。

## 6. 介面 / API（可選雲端元件）

僅在啟用雲端語意搜尋時需要。**所有端點都不接收或回傳密碼。**

| 方法 | 路徑 | 輸入 | 輸出 | 說明 |
|---|---|---|---|---|
| POST | `/index` | `{ id, search_text }` | `{ ok }` | 對非敏感 search_text 建立 / 更新向量索引 |
| POST | `/search` | `{ query }` | `{ ids: string[] }` | 語意檢索，僅回傳條目 ID |
| DELETE | `/index/:id` | — | `{ ok }` | 移除索引 |

注意：`search_text` 由 client 端在上傳前確保不含敏感欄位。查詢字串不放入 URL query string（避免敏感資訊外洩）。

## 9. 跨裝置端對端加密同步（E2EE Sync）

目標：使用者在任一裝置「登入帳號 + 輸入主密碼」即可取得同一份金庫，但**伺服器全程只持有密文**，永遠無法解讀內容（零知識）。核心觀念：能解密金庫的是**主密碼**，不是登入帳號；帳號只用來定位你的密文。

### 9.1 金鑰階層（Key Hierarchy）

關鍵設計是「**資料用一把隨機主金鑰加密；這把主金鑰再被多種方式包起來（wrap）**」。如此換主密碼或用復原碼救援時，只需重新包裝主金鑰，不必重新加密整個金庫。

```
主密碼 (master password, 只存在使用者腦中)
        │  Argon2id (KDF, 含 per-user salt)
        ▼
MEK  Master Encryption Key（由主密碼派生，不上傳、不落地）
        │  用 MEK 加密(wrap)
        ▼
VK   Vault Key（隨機產生的對稱金鑰，真正用來加解密金庫資料）
        │  用 VK 加密
        ▼
金庫資料（ServiceEntry…）→ ciphertext（AES-256-GCM）

額外：VK 同時被「復原碼派生的金鑰 RK」加密保存一份 → 見 §10
```

- **MEK**：主密碼經 Argon2id + 使用者專屬 salt 派生；僅存於本機記憶體（解鎖期間），永不上傳。
- **VK（Vault Key）**：建立帳號時隨機產生，是真正加解密金庫的金鑰。
- 伺服器儲存的是 `wrappedVK_byMEK`（被 MEK 包起來的 VK）與金庫密文；伺服器沒有 MEK，無法解開 VK，因此無法解密金庫。
- **換主密碼**：只需用舊 MEK 解開 VK，再用新 MEK 重新 wrap，上傳新的 `wrappedVK_byMEK`；金庫密文不必重算。

> 認證與加密分離：登入伺服器用的驗證值（如 SRP，或對主密碼再做一次 KDF 得到的 auth hash）與 MEK 各自獨立，伺服器即使存有 auth hash 也推不回主密碼或 MEK。

### 9.2 同步資料模型（伺服器端，全部為密文 / 非敏感）

```json
{
  "accountId": "user-uuid",
  "kdfParams": { "algo": "argon2id", "salt": "...", "memKiB": 65536, "iterations": 3, "parallelism": 1 },
  "wrappedVK_byMEK": "...",        // 被主密碼金鑰包起來的 Vault Key
  "wrappedVK_byRK":  "...",        // 被復原碼金鑰包起來的 Vault Key（§10）
  "items": [
    { "id": "abc123", "ciphertext": "...", "rev": 12, "updatedAt": 1750000000 }
  ],
  "vaultRev": 12                    // 整體版本號，供同步比對
}
```

伺服器看得到的只有：帳號 ID、KDF 參數、兩份被包起來的 VK、各條目的密文與版本資訊。看不到任何明文。

### 9.3 同步流程

**新裝置首次登入**

```
1. 輸入帳號 → 伺服器回傳 kdfParams + wrappedVK_byMEK
2. 輸入主密碼 → 本機 Argon2id 派生 MEK
3. 本機用 MEK 解開 wrappedVK_byMEK → 得到 VK
4. 下載 items[].ciphertext → 用 VK 解密 → 還原金庫到本地加密 DB
```

**日常同步（雙向）**

```
- Push：本地有變更的條目，用 VK 加密成 ciphertext，連同遞增 rev 上傳
- Pull：以 vaultRev 比對，下載遠端較新的條目，本機解密合併
- 全程只傳密文；MEK / VK 不離開本機
```

### 9.4 衝突合併策略

- **粒度**：以「條目（item）」為單位同步，降低衝突面。
- **預設策略**：條目級 **Last-Write-Wins**（以 `updatedAt` + `rev` 判定），多數個人單人多裝置情境已足夠。
- **保護機制**：偵測到並發修改同一條目時，保留落敗版本為一筆「衝突副本」而非直接覆蓋，避免資料遺失（對應 FR-20）。
- **可選升級**：未來資料量或協作需求提高時，再評估欄位級合併或 CRDT（列為開放問題）。

### 9.5 同步 API（皆為密文 / 非敏感，端點不接收明文密碼或主密碼）

| 方法 | 路徑 | 輸入 | 輸出 | 說明 |
|---|---|---|---|---|
| POST | `/account` | `{ kdfParams, wrappedVK_byMEK, wrappedVK_byRK, authVerifier }` | `{ accountId }` | 建立帳號（僅上傳包裝後金鑰與驗證值） |
| POST | `/auth` | `{ accountId, authProof }` | `{ token }` | 登入驗證（SRP / auth hash，不傳主密碼） |
| GET | `/vault` | `?since=rev` | `{ items[], vaultRev, wrappedVK_byMEK }` | 拉取密文與包裝後 VK |
| POST | `/vault/items` | `{ items: [{id, ciphertext, rev}] }` | `{ vaultRev }` | 推送密文變更 |
| PUT | `/account/rewrap` | `{ wrappedVK_byMEK }` | `{ ok }` | 換主密碼後更新包裝後 VK |

## 10. 帳號復原（忘記主密碼）

因為採端對端加密，**伺服器沒有主密碼也沒有 VK 的明文，無法「找回」主密碼**。復原的本質不是找回密碼，而是事先準備**另一把能解開 VK 的鑰匙**。對應 1Password 的 Recovery Code / Emergency Kit、Bitwarden 的 Emergency Access 與組織管理員重設。

### 10.1 復原碼機制（個人版主力方案）

建立帳號時：

```
1. 產生高強度隨機「復原碼」(recovery code，例如 128-bit，以易抄寫格式呈現)
2. 由復原碼經 KDF 派生 RK（Recovery Key）
3. 用 RK 加密 VK → wrappedVK_byRK，上傳伺服器
4. 提示使用者離線妥善保管復原碼（一次性顯示 / 可下載 Emergency Kit）
```

於是 VK 同時被兩把鑰匙包起來：`wrappedVK_byMEK`（主密碼）與 `wrappedVK_byRK`（復原碼）。

### 10.2 忘記主密碼的復原流程

```
1. 使用者選「忘記主密碼」，輸入帳號 + 復原碼
2. 伺服器回傳 kdfParams + wrappedVK_byRK
3. 本機由復原碼派生 RK → 解開 wrappedVK_byRK → 取回 VK
4. 使用者設定「新主密碼」→ 派生新 MEK → 以新 MEK 重新 wrap VK
5. 上傳新的 wrappedVK_byMEK（呼叫 /account/rewrap）；金庫密文不變
6. （建議）使新復原碼失效並重新產生一組，提示重新保管
```

整個過程伺服器只經手被包裝的金鑰，從未取得 VK 明文。

### 10.3 風險與告知（對應 NFR-08、FR-24）

- **無後門、無客服代救**：若主密碼與復原碼**同時遺失**，VK 無法解開，資料永久無法復原；設定流程須明確告知此取捨。
- **復原碼保管引導**：一次性顯示、提供可下載的 Emergency Kit、允許在已解鎖狀態重新產生（重新產生會更新 `wrappedVK_byRK`）。
- **軟性緩衝（非真正救援，但提升可用性）**：主密碼提示（hint，本地或加密保存）、在仍解鎖的其他裝置用 PIN / 生物辨識解鎖、定期匯出加密備份。

### 10.4 緊急聯絡人（可選，後續迭代，對應 FR-25）

以非對稱加密讓使用者預先把 VK（以對方公鑰加密）授權給受信任者；緊急時對方發起接管，經設定的**等待期**且未被拒絕後，可解密取得讀取 / 接管權限。授權可隨時撤銷。此項列為 M6。

> 設計哲學：使用者要的不是「LLM password manager」，而是「讓它感覺很聰明」。Password manager 的 domain 小且結構化，靠 normalization + alias + fuzzy + heuristic 就能做出「像 AI」的體驗。安全面的對應原則同樣務實：**用標準的金鑰包裝（key wrapping）+ 零知識同步 + 復原碼**，就能同時做到「換裝置登入就能用」與「密碼不以明文離開裝置」。
