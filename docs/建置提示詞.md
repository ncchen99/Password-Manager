# 建置提示詞 — 智慧化密碼管理器 PWA（給 AI 的指示）

> 將以下整段內容貼給負責開發的 AI（如 Claude Code / Cursor / 其他 coding agent）。它包含產品定位、技術選型、安全架構、前端設計原則與分階段任務。請 AI 嚴格遵守安全與設計約束。

## 0. 你的角色與目標

你是一位資深全端工程師暨產品設計師。請從零開始打造一個**現代感、易用、安全性極高、配色舒服**的智慧化密碼管理器 **PWA（Progressive Web App）**，手機與電腦共用同一套程式碼、可離線、可安裝。

最高指導原則：**密碼絕不以明文離開使用者裝置**。所有後端（Firebase）只接觸**密文**或**非敏感資料**，採零知識（zero-knowledge）架構。任何設計若與此衝突，以安全為優先。

請以「打造真實商用產品」的標準執行：可維護的程式碼結構、型別安全、清楚的錯誤處理、無障礙（a11y）、響應式。不要產生玩具等級的示範碼。

## 1. 產品概述

一個讓使用者「感覺很聰明」的密碼管理器，核心三件事：

1. **智慧化匯入**：使用者貼上一段格式雜亂的帳密文字，系統用「正規化 + 啟發式規則 + 有限狀態機」解析成結構化條目，並以「AI 猜測、使用者確認」的卡片流程匯入（不是全自動寫入）。
2. **智慧化搜尋**：用別名 / 模糊 / （可選）語意搜尋，輸入「臉書 / FB / Facebook」都能找到同一筆。
3. **端對端加密同步與復原**：換裝置登入 + 輸入主密碼即可使用同一份金庫；忘記主密碼可用復原碼救回。

重要觀念：這個 domain 很小且結構化，**不需要本地大型 LLM**。預設用 normalization + alias + fuzzy + heuristic 就能做出「像 AI」的體驗；語意搜尋只在需要時用「小型 embedding 模型」（非生成式）。

## 2. 技術選型（必須遵守）

### 2.1 後端：Firebase

- **使用 Firebase 作為後端**。本機已安裝 **Firebase CLI**，Firebase 設定檔已整理至專案根目錄的 `firebase/` 資料夾中。請直接從本機操作，並加上 `--config firebase/firebase.json` 指令（或在 `firebase/` 目錄下執行），例如：`firebase emulators:start --config firebase/firebase.json`。
- **Hosting**：以 **Firebase Hosting** 部署 PWA（含正確的 PWA headers、SPA rewrite、快取策略）。
- **Authentication**：用 **Firebase Auth** 管理帳號登入（Email 等）。注意：Firebase Auth 只負責「登入身分」，**不負責解密**——解密金鑰一律由主密碼在本機派生（見 §3）。
- **Firestore**：作為同步儲存，**只存密文與非敏感欄位**（被包裝的金鑰、各條目 ciphertext、版本號）。設計嚴格的 **Security Rules**：使用者只能讀寫自己 `uid` 底下的文件。
- **Cloud Functions**：在需要伺服器運算時使用——例如（可選）語意搜尋的 embedding 與向量檢索、或重運算的解析輔助。**Functions 端點一律不接收明文密碼、不接收主密碼**；只處理非敏感資料，且回傳搜尋結果時只回傳條目 ID。
- 善用 **Firebase Emulator Suite** 做本機開發與測試（Auth / Firestore / Functions / Hosting 模擬器）。

### 2.2 前端：PWA

- 框架：**React + TypeScript + Vite**（或同級現代框架），加上 PWA 能力（`vite-plugin-pwa` / Workbox：service worker、manifest、可安裝、離線快取 app shell）。
- **本地儲存**：瀏覽器端用 **IndexedDB**（建議透過 Dexie.js 之類封裝）保存**加密後的金庫**；解密只在記憶體中進行。（規格書原以 SQLite/SQLCipher 描述，PWA 環境改用 IndexedDB + WebCrypto 對等實作。）
- **加密**：使用 **WebCrypto API**——AES-256-GCM 加解密、Argon2id（或 WebCrypto PBKDF2 作為次選）做 KDF。金鑰只存記憶體，永不寫入明文、永不上傳。
- **UI 框架**：可使用 **Tailwind CSS + DaisyUI**。Icon 使用 **Heroicons**。若有專為 PWA 手機 App 設計、品質良好的 UI Kit 也可採用，但須與下方設計原則一致。

## 3. 安全架構（必須正確實作）

採標準的**金鑰包裝（key wrapping）**，讓「換裝置登入就能用」與「密碼不以明文離開裝置」並存：

```
主密碼 (master password，只在使用者腦中)
   │ Argon2id (+ per-user salt)
   ▼
MEK  Master Encryption Key（本機派生，不落地、不上傳）
   │ 用 MEK 包裝(wrap)
   ▼
VK   Vault Key（隨機對稱金鑰，真正用來加解密金庫）
   │ 用 VK 加密
   ▼
金庫資料 → ciphertext (AES-256-GCM)，存 IndexedDB / Firestore

額外：VK 另以「復原碼派生的 RK」包裝一份 → wrappedVK_byRK（供復原）
```

實作要點：

- Firestore 只存：`kdfParams`、`wrappedVK_byMEK`、`wrappedVK_byRK`、各條目 `{ id, ciphertext, rev, updatedAt }`、`vaultRev`。**絕不存** password / MEK / VK 明文。
- **登入身分**（Firebase Auth）與**解密金鑰**（主密碼→MEK）分離；伺服器即使被攻破也無法解密金庫。
- **換主密碼**：用舊 MEK 解出 VK，再用新 MEK 重新包裝，只更新 `wrappedVK_byMEK`，金庫密文不必重算。
- **復原碼**：建立帳號時產生高強度隨機復原碼（如 128-bit，易抄寫格式），派生 RK 包裝 VK 為 `wrappedVK_byRK`，提示使用者離線保管（一次性顯示 + 可下載 Emergency Kit）。
- **忘記主密碼流程**：輸入帳號 + 復原碼 → 派生 RK → 解開 `wrappedVK_byRK` 取回 VK → 設新主密碼 → 重新包裝上傳。
- **明確告知**：無後門、無客服代救；主密碼與復原碼同時遺失即資料無法復原。
- 一律 TLS；敏感資料不放入 URL query string；同步衝突採條目級 Last-Write-Wins，落敗版本保留為「衝突副本」避免遺失。

## 4. 智慧化匯入（解析管線）

請實作下列純前端（不需 LLM）的管線：

1. **Step 0 正規化（Canonicalization）**：欄位同義詞字典（acc/account/帳號/user→username；pwd/pass/password/密碼→password）、大小寫正規化、全形↔半形（`：`→`:`）、統一分隔符（`：- = > 】 |` → `:`）。
2. **Step 1 逐行分類**：以 regex 將每行判為 service / username / password / unknown。
3. **Step 2 特徵推論（Heuristic Scoring）**：無明確關鍵字時依特徵打分（含 @ → username；09 開頭手機 → username；含 `.com` → service/url；大小寫+數字混合高 entropy → password），保留 confidence。
4. **Step 3 有限狀態機分組**：組成「一個 service + 多組 credentials」；遇空行 commit 一組；username/password 皆可為空（允許只有密碼）。
5. **Step 4 確認 UI（AI Assist, not Auto）**：解析結果以可編輯卡片呈現，使用者逐筆確認/修改後才寫入。目標正確率 85%+，其餘由使用者修正。

資料模型（解密後）：`ServiceEntry { id, service, aliases[], url?, tags[], credentials: Credential[] }`，`Credential { username?, password?, otp?, note? }`。

## 5. 智慧化搜尋

- **預設（必做）**：別名 + 模糊搜尋。對 service / aliases / url / tags 做 lowercase、去空白、trigram / Levenshtein 比對；別名字典可自動產生（由 domain 推導）並可手動維護。資料量數十～數千筆，純前端即可即時。
- **可選增強**：語意搜尋用小型 embedding 模型（MiniLM / bge-small / e5-small，量化後數十 MB），僅對非敏感 `search_text` 建立向量。可在瀏覽器內或 Cloud Functions 運算；若走 Functions，**只回傳條目 ID**，前端再用 ID 在本地解密取出密碼。

## 6. 前端設計原則（打造真實商用產品）

整體目標：**簡潔俐落、一目瞭然、易於使用**，具現代感與舒服配色。請把以下當作硬性設計規範。

### 6.1 視覺與配色

- 採克制、專業的配色：以中性底色 + 一個主色（accent）為主，避免花俏。提供**淺色與深色模式**，皆需符合 WCAG AA 對比。
- 留白充足、層次清楚；字級階層分明（標題 / 內文 / 次要說明）。
- 動效克制：用於回饋與轉場，避免炫技；尊重 `prefers-reduced-motion`。

### 6.2 版面與佈局（重要）

- **手機頁面減少使用卡片（Cards）**，改用**水平分割線（horizontal dividers）**區隔列表項目，讓清單更輕、更一目瞭然。卡片僅在真正需要強調的群組才用。
- 列表型畫面（密碼條目清單）以「列 + 分隔線」呈現，每列：服務名稱、次要資訊（username 預覽 / tags），右側操作。
- 重要操作（新增、匯入）放在容易單手觸及的位置；手機底部可用固定動作列。

### 6.3 元件與手機適配（重要）

- 可使用 **DaisyUI**（基於 Tailwind）作為元件基礎，但**所有元件都要做手機端適配**。
- 桌面上的小型浮動元件（如 popover / dropdown / tooltip）在手機上**必須放大改用 Modal 或 Bottom Sheet**呈現，確保觸控目標夠大、好操作。
- 觸控目標至少 44×44px；表單欄位、按鈕在手機上加大。
- Icon 一律使用 **Heroicons**（風格統一、line/solid 一致使用）。
- 若採用專為 PWA 手機設計的 UI Kit，須與上述原則一致，不得引入過重或風格不符的元件。

### 6.4 無障礙與可用性

- 語意化 HTML、鍵盤可操作、focus 樣式清楚、ARIA 標註正確。
- 載入 / 空狀態 / 錯誤狀態都要有明確設計，文案友善。
- 安全相關操作（顯示密碼、刪除、復原）需有清楚確認與回饋。

## 7. 工作方式與分階段任務

請按里程碑推進，每階段可獨立執行與測試，並善用 Firebase Emulator：

- **M1 — 專案骨架 + 本地 MVP**：建立 Vite + React + TS + Tailwind + DaisyUI + PWA 設定；`firebase init`（Hosting / Auth / Firestore / Functions / Emulators）；IndexedDB + WebCrypto 加密儲存；手動新增條目；別名 + 模糊搜尋；依設計原則完成清單 UI（分割線、深淺色）。
- **M2 — 智慧匯入**：完成 §4 的解析管線與確認 UI。
- **M3 — E2EE 同步 + 復原碼**：金鑰階層、Firestore 密文同步、Security Rules、復原碼與忘記主密碼流程（§3）。
- **M4 — 語意搜尋（可選）**：小型 embedding，瀏覽器內或 Cloud Functions，只回傳 ID。
- **M5 — 打磨**：a11y、效能、離線、安裝體驗、錯誤處理、測試。

開始前，請先輸出：

1. 你建議的**完整技術堆疊與資料夾結構**。
2. **Firestore 資料結構與 Security Rules** 草案。
3. **金鑰階層與加解密流程**的實作計畫（用哪些 WebCrypto API）。
4. M1 的具體步驟與將建立的檔案清單。

確認方向後再開始寫程式。過程中遇到安全或設計取捨時，明確說明選項與你的建議，並以「密碼不以明文離開裝置」為最高原則。

## 8. 驗收檢查清單（Definition of Done）

- [ ] 密碼明文與金鑰從未上傳、未寫入 Firestore、未進 URL；可由網路面板與 Firestore 內容驗證。
- [ ] Firestore Security Rules 限制使用者只能存取自己的資料，並通過測試。
- [ ] 換裝置登入 + 輸入主密碼可取回完整金庫；忘記主密碼可用復原碼救回。
- [ ] 匯入解析在常見格式達 85%+ 正確率，且一律經使用者確認才寫入。
- [ ] 別名 / 模糊搜尋可用「臉書 / FB / Facebook」找到同一筆。
- [ ] PWA 可安裝、可離線使用核心功能；Lighthouse PWA / a11y / 效能分數良好。
- [ ] 手機端以分割線取代卡片；桌面小浮層在手機改為 Modal / Bottom Sheet；Heroicons 一致；深淺色皆達 WCAG AA。
