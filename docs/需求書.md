# 智慧化密碼管理器 — 需求書 (Requirements Document)

**版本**：v1.1
**日期**：2026-06-15
**目標平台**：PWA（手機與電腦共用同一套網頁應用）
**本版變更**：新增「跨裝置端對端加密同步（E2EE Sync）」與「忘記主密碼的復原碼機制」需求。

---

## 1. 專案背景與目標

使用者想自製一套密碼管理器，核心訴求是「讓它感覺很聰明」，但又不願意為此付出兩個代價：

1. 把密碼送上雲端（隱私風險）。
2. 在手機端塞一個數 GB 的本地大型語言模型（佔空間、耗效能）。

本專案的設計原則是：**用最小的技術成本，做出「像 AI」的體驗**。密碼管理器的資料 domain 很小、結構很固定，因此大部分「聰明」的效果可以靠正規化、啟發式規則、模糊比對與小型 embedding 模型達成，不需要完整的生成式 LLM。

核心目標：

- 使用者可以**貼上一整段雜亂的帳密文字**，系統自動整理成結構化資料並存入本地。
- 使用者可以用**別名 / 模糊 / 語意搜尋**（例如「臉書」「FB」「Facebook」都能找到同一筆）找到帳密。
- **密碼永遠不以明文離開裝置**；任何雲端運算都只接觸非敏感資料或無法解讀的密文。
- 使用者可以在**任何裝置登入並輸入主密碼後使用同一份金庫**（跨裝置端對端加密同步）。
- 即使**忘記主密碼**，也能透過事先保管的**復原碼**重新取回金庫。

---

## 2. 問題拆解

整個需求其實是兩個彼此獨立的問題，應分開設計：

| 問題 | 說明 | 難度 |
|---|---|---|
| 智慧化匯入（Parsing / Structuring） | 使用者貼上一段文字，系統辨識出 service、username、password、otp、note、url 等欄位 | 較難（格式不固定） |
| 智慧化搜尋（Semantic Retrieval） | 搜尋「FB」找到 Facebook、搜尋「Google 信箱」找到 Gmail | 較易（搜尋空間小） |

搜尋比匯入容易得多，且兩者可用完全不同技術解決。

---

## 3. 目標使用者與使用情境

**主要使用者**：希望自行掌控資料、重視隱私、有一定數量帳密需要管理的個人。

**典型使用情境**：

1. **批次匯入**：使用者從舊筆記、記事本或截圖整理出一大段帳密文字，一次貼上匯入。
2. **快速查找**：使用者要登入某服務時，用任何聯想得到的關鍵字（中文、英文、縮寫）找出帳密。
3. **多組帳號**：同一個服務（如 ProtonVPN）底下有多組帳號密碼。
4. **不完整資料**：某些項目只有密碼沒有帳號，或只有一組數字（可能是手機號當帳號）。

## 4. 使用者故事 (User Stories)

- US-01：身為使用者，我想**貼上一段格式雜亂的帳密文字**，讓系統自動整理成一筆筆條目，這樣我就不必手動逐欄輸入。
- US-02：身為使用者，我想在匯入時**先看到系統的解析結果並逐筆確認 / 修正**，再正式存入，以避免匯入錯誤。
- US-03：身為使用者，我想用**中文、英文或縮寫別名**搜尋（臉書 / FB / Facebook），都能找到正確條目。
- US-04：身為使用者，我想讓**同一個服務底下保存多組帳密**，而不必拆成多筆不相關的條目。
- US-05：身為使用者，我想匯入**只有密碼、沒有帳號**的條目，系統不該強迫我補齊欄位。
- US-06：身為使用者，我希望**密碼絕不上傳雲端**，即使我使用了雲端搜尋功能。
- US-07：身為使用者，我想在**手機和電腦上使用同一套 PWA**，體驗一致。
- US-08：身為使用者，我想在**換新裝置上登入帳號並輸入主密碼後，就能看到我所有的密碼**，不必手動搬移。
- US-09：身為使用者，我想確保即使啟用了雲端同步，**伺服器也讀不到我的密碼明文**，因為解密只在我輸入主密碼後於本機進行。
- US-10：身為使用者，萬一**忘記主密碼**，我想用事先保管好的**復原碼**重設主密碼並救回金庫，而不是整個資料報廢。
- US-11：身為使用者，我希望在多台裝置都修改了金庫時，系統能**合理合併變更**，不要讓我遺失資料。

## 5. 功能需求 (Functional Requirements)

### 5.1 智慧化匯入

- FR-01：支援使用者一次貼上多筆、多行的帳密文字。
- FR-02：**輸入正規化（Canonicalization）**：在解析前統一格式 —
  - 欄位關鍵字同義詞對應（acc / account / 帳號 / 帳戶 / user → username；pwd / pass / password / 密碼 → password）。
  - 全形 / 半形轉換（全形冒號「：」→ 半形「:」；全形連字號 → 半形）。
  - 統一分隔符號（: - = > 】 | 等 → 統一成 `:`）。
  - 大小寫正規化（關鍵字轉小寫）。
- FR-03：**逐行分類（Line Classification）**：判斷每行屬於 service / username / password / unknown。
- FR-04：**狀態機分組（Grouping）**：以有限狀態機將多行組成「一個 service + 多組 credentials」的結構；遇空行 commit 一組 credential。
- FR-05：**容錯 schema**：username 與 password 皆為選填，允許只有密碼、只有帳號的條目。
- FR-06：**特徵推論（Heuristic Inference）**：當沒有明確關鍵字時，依特徵推測欄位（email/phone 偏 username；高 entropy、含大小寫數字混合偏 password；含 `.com` 偏 service/url），並給出 confidence score。
- FR-07：**匯入確認 UI（AI Assist, not AI Auto）**：解析結果以可編輯卡片呈現，使用者逐筆確認 / 修改後才寫入資料庫。
- FR-08：解析正確率目標達 85% 以上（在常見格式下），其餘由使用者於確認步驟修正。

### 5.2 智慧化搜尋

- FR-09：每筆條目可儲存 service 名稱、別名（aliases）、URL、tags、note。
- FR-10：支援**模糊搜尋**：lowercase、去空白、trigram、Levenshtein 比對。
- FR-11：支援**別名字典**自動 / 手動產生（facebook → fb, 臉書；instagram → ig, 哀居 等）。
- FR-12：（可選）支援**語意搜尋**：以小型 embedding 模型對「非敏感欄位」建立向量索引。
- FR-13：搜尋結果即使透過 Server 處理，也只回傳條目 ID，不回傳任何密碼。

### 5.3 資料保存與安全

- FR-14：本地資料以 AES 加密儲存（SQLCipher / 加密 payload）。
- FR-15：解密金鑰僅存在於本地，不上傳。
- FR-16：上傳至 Server 的資料（若啟用雲端搜尋）僅限非敏感欄位（service、aliases、url、tags 的 embedding 或索引），且密碼、otp、note 一律不上傳；username 可選擇不上傳或雜湊化。

### 5.4 跨裝置端對端加密同步（E2EE Sync）

- FR-17：使用者可建立帳號並設定**主密碼（master password）**；主密碼用於在本機派生加解密金鑰，**永不上傳**伺服器。
- FR-18：金庫資料在離開裝置前，一律先在本機加密成**密文（ciphertext）**；同步至雲端的只有密文，伺服器無法解讀內容。
- FR-19：使用者在任一新裝置登入帳號並輸入主密碼後，可下載密文並於本機解密，取得完整金庫。
- FR-20：同步須支援**多裝置變更合併**，盡量避免資料遺失；發生衝突時應有可預期的處理規則（見規格書）。
- FR-21：使用者可選擇**只用本地（關閉同步）**；關閉同步時核心功能仍完整可用（呼應漸進式增強原則）。

### 5.5 帳號復原（忘記主密碼）

- FR-22：系統須提供**復原碼（recovery code）/ 復原金鑰**機制：建立帳號時產生一組高強度隨機碼，提示使用者離線妥善保管。
- FR-23：忘記主密碼時，使用者可用復原碼驗證並**重設新主密碼**，重新取回金庫（其原理為：金庫主金鑰另以復原碼派生之金鑰加密保存一份備援）。
- FR-24：系統須明確告知使用者：**伺服器沒有主密碼與金庫明文，無法代為找回**；若主密碼與復原碼同時遺失，資料將無法復原。
- FR-25（可選，後續）：支援**緊急聯絡人（emergency access）**，由使用者事先以加密方式授權受信任者，於緊急情況經等待期後取得接管權限。

## 6. 非功能需求 (Non-Functional Requirements)

- NFR-01（隱私）：密碼、解密金鑰、敏感備註在任何情況下都不離開使用者裝置。
- NFR-02（效能 / 空間）：避免在裝置上部署數 GB 的生成式 LLM；若使用本地模型，僅限量化後數十 MB 等級的 embedding 模型。
- NFR-03（效能 / 搜尋）：資料量通常為數十至數千筆，SQLite + FTS + 模糊搜尋即可達到即時回應。
- NFR-04（可用性）：匯入流程以「猜測 + 確認」為核心，提高容錯率與使用者信任。
- NFR-05（跨平台）：以 PWA 形式提供，手機與電腦共用同一程式碼與體驗，支援離線使用。
- NFR-06（漸進式 AI）：AI 能力（語意搜尋、雲端解析）為可選增強，關閉時核心功能仍完整可用。
- NFR-07（零知識 / 端對端加密）：同步伺服器採零知識架構——僅持有密文，無主密碼、無金庫明文、無解密金鑰，無法解讀或復原使用者資料。
- NFR-08（復原責任）：因採端對端加密，無後門、無客服代救；復原碼 / 緊急工具包的保管責任在使用者端，產品須在設定時清楚說明此取捨。
- NFR-09（傳輸安全）：所有同步流量以 TLS 傳輸；敏感資料不放入 URL query string。

## 7. 範圍界定

**包含（In Scope）**

- 智慧化匯入（正規化 + 啟發式 parser + 確認 UI）。
- 別名 / 模糊搜尋；本地加密儲存。
- 可選的小型 embedding 語意搜尋與隱私保護的雲端索引。
- 跨裝置端對端加密同步（E2EE Sync）。
- 復原碼 / 復原金鑰機制（忘記主密碼的復原流程）。

**暫不包含（Out of Scope，可列為後續）**

- 完整的生成式 LLM 本地部署。
- 緊急聯絡人（emergency access）授權接管（列為後續迭代，FR-25）。
- 瀏覽器自動填表、團隊 / 組織共享與管理員重設、二步驟驗證碼產生器等進階功能（除非後續另立需求）。
